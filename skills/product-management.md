# Product Management

**Scope:** Brainstorming, requirements discovery, user stories, prioritisation.

## Rules
- Start with the problem, not the solution — articulate the user pain before proposing features
- Every feature needs a clear "who benefits and how" statement
- Separate must-haves from nice-to-haves explicitly (MoSCoW or similar)
- Identify assumptions and flag them — don't let unvalidated assumptions become requirements
- Think in user flows, not screens — map the journey end-to-end before detailing UI
- Edge cases are requirements too — enumerate failure states, empty states, permission boundaries

## Brainstorming
- Explore at least 3 approaches before converging on one
- Challenge each approach: what breaks, what scales, what's the simplest version
- Identify constraints early: technical, timeline, team, dependencies
- Ask "what's the cheapest experiment that validates this?" before committing to full build

## Requirements Output
- User stories in "As a [role], I want [action], so that [outcome]" format
- Acceptance criteria per story — testable, unambiguous
- Out-of-scope section — explicitly state what this does NOT include
- Dependencies on other teams, services, or data

## Prioritisation
- Rank by impact vs effort — high impact + low effort first
- Group into milestones or phases with clear deliverables per phase
- Identify the critical path — what blocks everything else
- Flag risks and mitigations for each milestone

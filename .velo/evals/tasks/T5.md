# T5 — Ambiguity catch (anti-pattern test)

## Command

`/velo:new`

## Prompt

> Make the spec-check faster.

## Setup

None.

## Expected behavior

- PM or Velo immediately surfaces the ambiguity: "Faster relative to what baseline? What metric — wall time, tokens, agent count? What's the target threshold?"
- The system pauses and waits for clarification — does NOT proceed to write a PRD, EDD, or any code.
- If the team somehow proceeds past PM, spec-checker should return BLOCKED on the resulting PRD because the acceptance criteria are unmeasurable.

## Anti-patterns to flag

- PM writes a PRD with invented acceptance criteria ("response time under 500ms" — not in the brief).
- TL produces an EDD without challenging the brief.
- Build phase fires.
- Code is committed.

## Scoring notes

This is the only task scored as **pass/fail** (the T5-specific dimension in the rubric). The 5 numeric dimensions still apply if you want to track them, but the headline result is binary: did the system catch the ambiguity, or did it execute a bad brief?

## External Review

### Critical (must fix before build)
- [Read-only safety is not actually enforceable]: Step 4 allows `Read/Grep/Glob/Bash`, and S3 says to spawn a “read-only” agent via the generic Agent tool. In this repo, [TEAM.md](/Users/rajasekarm/Documents/focus/velo/TEAM.md:58) defines agent spawning only as prompt text, with no permission model, and [commands/yo.md](/Users/rajasekarm/Documents/focus/velo/commands/yo.md:16) shows “no code” is a behavioral rule, not a hard constraint. As written, hunt can be pushed into file writes, destructive shell, or secret leakage from configs/logs. Suggested fix: make MVP tool access explicit and narrow. Default to `Read`/`Grep`/`Glob` only, ban `Bash` except for a short read-only whitelist, and either remove subagents from v1 or specify a real constrained subagent contract with secret-redaction rules.
- [The state machine is underspecified and can drift or loop]: D3/D4 plus Step 4/F1/S1 define counters and reranks, but they do not define when `Hypothesis.status` changes, when `noProgressCounter` and `stepsOnActive` reset, how exhausted hypotheses are replaced, or what “Keep investigating” means after Step 5 already confirmed root cause. That creates a real risk that the printed Hunt board diverges from the actual in-memory state, or that the loop continues indefinitely on shallow “confidence” changes. Suggested fix: add an explicit transition table and invariants. The board should be a pure render of `HuntState`, status changes should be deterministic, reset rules should be explicit, and opening a new hypothesis cycle after F1 should require an explicit user choice.
- [Boundary routing will misclassify common debugging requests]: Step 1 routes on imperative verbs and “pure conceptual question,” but PRD US-8 requires routing on observed defect, repro/near-repro, and whether root cause is already known. Real hunt invocations are often phrased as “fix login 500” or “debug retry loop,” which Step 1.3 would wrongly push to `/velo:task` or `/velo:new`. The reverse gap also exists: there is no explicit route for “I already know the root cause, now I want the fix,” which should go to `/velo:task`. Suggested fix: replace verb heuristics with a 3-way classifier: defect + unknown cause => `/velo:hunt`; discussion/expected behavior => `/velo:yo`; known cause / ready to change code => `/velo:task` or `/velo:new`.

### Significant (should fix)
- [US-7 is only partially implemented]: S3 captures the trigger and announcement, but not the prompt, output shape, merge-back behavior, or enforcement needed to make a subagent truly read-only. Suggested fix: either cut US-7 from v1 or define the exact agent contract: allowed tools, max file/log fan-out, expected findings schema, and how those findings update HuntState.
- [The caps are likely too aggressive for real bugs]: One “step” is a single read/grep/log action. With that granularity, “3 consecutive no-progress steps” and “5 steps per hypothesis” will fire before many real investigations have even scoped the problem, especially env-diff, config, flaky repro, and trace-following cases. Suggested fix: measure progress at the level of an investigation bundle, not every individual read, and make the caps soft thresholds with explicit override rather than hard stops.
- [Failure modes miss several realistic dead ends]: F1-F7 and S1-S3 do not cover tool failure, missing permissions, stack traces with no first-party frame, flaky/non-deterministic repros, contradictory evidence that should split or merge hypotheses, or secret-bearing artefacts that should not be echoed. Suggested fix: add operational failure modes and handling rules for “cannot access artefact,” “repro is flaky,” “no first-party frame,” and “evidence contradicts active hypothesis.”
- [Cross-command dependencies are brittle]: D6 says to reuse yo Step 7, but that contract currently lives only in [commands/yo.md](/Users/rajasekarm/Documents/focus/velo/commands/yo.md:320). `/velo:task` in [commands/task.md](/Users/rajasekarm/Documents/focus/velo/commands/task.md:17) accepts freeform task text and does not define a formal handoff brief schema. That means hunt can drift silently as yo/task evolve. Suggested fix: define a shared handoff contract or partial for AskUserQuestion options and brief shape, and have all three commands reference that shared contract.
- [PRD coverage is incomplete on the boundary stories]: US-1/2/3/5 are mostly mapped to Steps 1-5 and 7. US-4 is only partially concrete because the task/new handoff brief content is underspecified. US-6/8 are partial because the positive entry criteria for hunt are never codified as “specific observed defect + repro/near-repro + unknown cause,” and F3/F4 blur diagnosis routing versus implementation routing. Suggested fix: add a PRD traceability section that maps each US and acceptance criterion to exact steps/decisions/failure rules.

### Minor (worth noting)
- [The Hunt board render itself is not specified]: The state types are present, but there is no canonical board format, field ordering, or evidence display rule. Two implementers will render it differently. Suggested fix: add one example board and require all summaries to be generated from `HuntState`.
- [Documentation updates should be mandatory, not optional]: The EDD makes `README.md` conditional, but discoverability also depends on [WORKFLOW.md](/Users/rajasekarm/Documents/focus/velo/WORKFLOW.md:1), which currently documents only `/velo:new`, `/velo:task`, and `/velo:yo`. Suggested fix: require updates to both command index docs.
- [“Reuse yo Step 7 verbatim” is inaccurate]: hunt wants a different option set from yo’s current handoff in [commands/yo.md](/Users/rajasekarm/Documents/focus/velo/commands/yo.md:336). Suggested fix: say “reuse the same AskUserQuestion mechanism” and define hunt’s own option contract explicitly.

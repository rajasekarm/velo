## External Review

### Critical (must fix before build)
- Failure is not representable cleanly at the phase or task level. `AgentEntry` can fail, but `Phase.status` only allows `pending | in_progress | completed | skipped`, and there is no top-level task lifecycle state. A run can therefore show a failed BE agent while the task still appears merely "in progress" or even "completed" in summaries. That breaks G1/G2 for failed workflows. Add `task.status` (`active | completed | failed | cancelled`) and add `failed` to `Phase.status`, then define how `current_phase` behaves after a failure. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:117), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:139), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:301).
- The write protocol is unsafe for the parallel execution model Velo already has. `new.md` and `task.md` run builders and reviewers in parallel, but the design does not specify atomic writes, read-modify-write against the latest file, or who owns phase-completion writes. That creates lost-update races and transient corrupt JSON during the core use case of checking status mid-run. Require atomic temp-file-and-rename writes, always reload and merge the latest `status.json` before writing, and make the orchestrator perform phase completion only after all agents in that phase have joined. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:280), [commands/new.md](/Users/rajasekarm/Documents/focus/velo/commands/new.md:113), [commands/task.md](/Users/rajasekarm/Documents/focus/velo/commands/task.md:36).
- Agent identity and ordering do not satisfy the PRD. The protocol appends an agent entry before spawn and later updates "the last entry with that name and status `in_progress`." That gives start order, not completion order, and it becomes ambiguous as soon as the same agent can be retried or ever run concurrently. The PRD explicitly wants chronological completion order. Add a stable `run_id` per invocation and define rendering order explicitly, preferably by `completed_at` with in-progress entries grouped separately. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:148), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:293).
- The naming contract is already inconsistent. The design uses `engineering-doc` / `engineering-doc-approval`, while the PRD and artifact naming use `engineering-design-doc` / `engineering-design-doc-approval`. In a markdown-command system, that is a real contract bug, not a cosmetic one. Publish one authoritative table of canonical phase and gate strings and make every schema example, protocol step, and renderer use it. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:55), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:133), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:268).

### Significant (should fix)
- The persisted file has no `schema_version`. This is an on-disk contract intended to survive interruptions and future releases. Without versioning, even small schema changes later become guesswork or silent breakage. Add `schema_version: 1` now and require `velo:status` to reject or degrade gracefully on unknown versions.
- The design uses multiple competing definitions of "latest": filesystem mtime for default task selection, `task.updated_at` for stale detection, and rendered update times from the JSON. Those values can diverge. Pick one source of truth for selection, sorting, and staleness, preferably the JSON field, and use filesystem mtime only as a fallback. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:45), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:16), [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:287).
- The CLI contract is underspecified for flag combinations and machine output. `--list --json` is undefined, mutual exclusivity is only partially documented, and the JSON output shape for list mode is absent. Define the accepted combinations and exact output schemas, or reject unsupported combinations explicitly. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:41).
- Committing `status.json` by default is weakly justified. Disk persistence already satisfies G3. Putting runtime state into source control adds noisy diffs, branch-merge conflicts, and operational metadata mixed with code artifacts. Unless "audit trail in git" is a real product requirement, this should default to local task state with an opt-in policy for teams that want it. See [engineering-design-doc.md](/Users/rajasekarm/Documents/focus/velo/.velo/tasks/velo-status-command/engineering-design-doc.md:14).

### Minor (worth noting)
- `skipped` phases have no reason field. For an audit feature, "skipped because no schema changes" is materially more useful than a bare status bit.
- `error` is defined as a one-line string, but there is no normalization rule. Tool errors are often multiline or very long. Define truncation/sanitization so the file stays readable and the terminal view stays stable.

### Verdict
REVISE

Most of the PRD is mapped: US-1, US-3, US-4, US-5, US-6, and US-7 all have clear design hooks. The material misses are failure-state modeling, concurrent write correctness, and the US-2/E7 requirement around completion ordering. Those are design-level gaps, not implementation details.
